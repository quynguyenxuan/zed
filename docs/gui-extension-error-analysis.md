# GUI Extension Error Analysis

## Error Message
```
Failed to load extension: gui-test
Loading extension from "/home/pquy/.local/share/zed/extensions/installed/gui-test":
loading wasm extension: gui-test: failed to instantiate wasm extension:
component imports instance `zed:extension/gui`, but a matching implementation
was not found in the linker: instance export `set-view` has the wrong type:
function implementation is missing
```

## Root Cause Analysis

### 1. What's Happening

The extension **compiles successfully** but **fails to load** at runtime because:
- The WASM component expects `zed:extension/gui` interface with `set-view` function
- The Wasmtime linker doesn't have the implementation wired up
- Even though `impl gui::Host for WasmState` exists, it's not connected to the linker

### 3. The Problem: Bindgen Configuration

**File**: `crates/extension_host/src/wasm_host/wit/since_v0_9_0.rs:13-33`
```rust
wasmtime::component::bindgen!({
    async: true,
    trappable_imports: true,
    path: "../extension_api/wit/since_v0.9.0",
    world: "panel-ui",
    with: {
        "worktree": super::since_v0_8_0::ExtensionWorktree,
        "project": super::since_v0_8_0::ExtensionProject,
        "key-value-store": super::since_v0_8_0::ExtensionKeyValueStore,
        "zed:extension/common": super::since_v0_8_0::common,
        "zed:extension/lsp": super::since_v0_8_0::lsp,
        "zed:extension/http-client": super::since_v0_8_0::http_client,
        "zed:extension/github": super::since_v0_8_0::github,
        "zed:extension/process": super::since_v0_8_0::process,
        "zed:extension/platform": super::since_v0_8_0::platform,
        "zed:extension/nodejs": super::since_v0_8_0::nodejs,
        "zed:extension/context-server": super::since_v0_8_0::context_server,
        "zed:extension/dap": super::since_v0_8_0::dap,
        "zed:extension/slash-command": super::since_v0_8_0::slash_command,
        // ❌ MISSING: "zed:extension/gui": ...
    }
});
```

**Issue**: The `with` clause maps old interfaces to existing implementations from `since_v0_8_0`, but `gui` is NEW in v0.9.0 and has NO mapping.

### 4. Why This Causes the Error

1. **Build Phase**:
   - Bindgen generates code for `panel-ui` world
   - It sees `import gui` in `panel-ui.wit`
   - It generates `gui` module and `gui::Host` trait
   - ✅ Everything compiles

2. **Runtime Phase**:
   - Extension loads and Wasmtime validates component
   - Component declares: "I import `zed:extension/gui` with `set-view` function"
   - Wasmtime checks linker: "Do you have `zed:extension/gui::set-view`?"
   - Linker checks its registry: ❌ Not found!
   - **Reason**: `PanelUi::add_to_linker()` was generated by bindgen, but because `gui` has no `with` mapping, the generated code might not properly wire it up

### 5. Comparison with Working Modules

All other modules have explicit mappings:
```rust
"zed:extension/lsp": super::since_v0_8_0::lsp,
```
This tells bindgen: "Don't generate new code for LSP, reuse the existing one from v0.8.0"

But `gui` is NEW, so it should:
- Either be generated fresh (no `with` entry needed)
- Or have proper module path if it exists elsewhere

### 6. Potential Causes

#### Hypothesis A: Missing Module Path
If `gui` module is defined elsewhere (e.g., in a separate file), it needs to be referenced:
```rust
"zed:extension/gui": super::since_v0_9_0::gui,
```

#### Hypothesis B: Bindgen Bug/Issue
The generated `PanelUi::add_to_linker()` might not include `gui::add_to_linker()` call.

#### Hypothesis C: Namespace Collision
The `gui` interface name might conflict with something, causing bindgen to skip it.

### 7. How Wasmtime Component Model Works

```
Component declares:    (import "zed:extension/gui" (instance ...))
                                                         ↓
Linker must provide:   linker.instance("zed:extension/gui")
                                                         ↓
                       Must contain:  func "set-view" ...
                                                         ↓
                       Wired to:      WasmState::set_view()
```

If any link is broken, you get the error we see.

If `gui` is defined in a separate module, add to `with`:


### Solution 3: Check for Module Name Collision

The bindgen might be generating `gui` under a different path:

```rust

// Explicitly re-export gui if needed:

```

### Solution 4: Manual Linker Setup (Last Resort)

If bindgen doesn't wire it up automatically, manually add:

```rust

    static LINKER: OnceLock<Linker<WasmState>> = OnceLock::new();
    LINKER.get_or_init(|| {
        let mut linker = super::new_linker(executor, PanelUi::add_to_linker);

        // Manually add GUI interface if missing
        gui::add_to_linker(&mut linker, |state| state).unwrap();

        linker
    })
}
```

## Conclusion

The issue is in the **wasmtime bindgen configuration** for v0.9.0. The `gui` interface is defined in WIT and implemented in Rust, but the bindgen-generated code doesn't properly wire it into the linker, causing the runtime error.

The fix requires either:
1. Adjusting the `with` clause in bindgen config
2. Manually ensuring `gui` module is properly exported
3. Verifying `PanelUi::add_to_linker()` includes `gui::add_to_linker()`
